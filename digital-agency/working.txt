Digital Agency Project Working Notes
===================================

Overview
--------
This project is a PHP + MySQL digital agency web application with:
- Public marketing pages
- User/client portal
- Admin panel
- Order management
- Chat and negotiation flow
- Offer and milestone payment support
- File uploads for chat, payments, services, team, and progress

Negotiation / Marketplace Flow
------------------------------

1. Client opens `user/services.php`
2. Client clicks request button for a service
3. `user/order-service.php` creates:
   - order request
   - first chat message
4. Client continues conversation in `user/chat/inbox.php`
5. Admin opens `admin/chat/send.php`
6. Admin replies and can send:
   - full payment offer
   - milestone payment offer
7. Client can:
   - accept offer
   - reject offer
   - counter offer
8. If accepted, client goes to `user/pay-order.php`
9. Client uploads payment proof
10. Admin reviews order/payment and continues work
11. Admin updates progress from `admin/orders/add-progress.php`
12. Client tracks progress in `user/order-progress.php`

CODE PART DETAILS AND HOW THEY WORK
============================================================

1. `includes/config.php`
------------------------

Main code purpose:
- Detect local vs live environment
- Set constants used everywhere

Important code behavior:
- Checks `$_SERVER['HTTP_HOST']`
  - if localhost / 127.0.0.1:
    - uses local DB settings
    - sets `BASE_URL` to `/digital-agency`
  - otherwise:
    - uses live hosting DB settings
    - sets live `BASE_URL`

Why it matters:
- Every page depends on these constants for database connection, session names, upload paths, and links.


2. `includes/db.php`
--------------------

Main code purpose:
- Open MySQL connection
- Make connection globally available as `$conn`
- Load schema helper

Important code behavior:
- Calls `mysqli_connect(...)`
- Stops execution with `die(...)` if database connection fails
- Calls `mysqli_set_charset($conn, "utf8mb4")`
- Requires `includes/marketplace.php`
- Calls `ensureMarketplaceSchema($conn)`

Why it matters:
- This file is the bootstrap for almost all database-driven pages.
- The negotiation/milestone feature depends on the schema auto-update call here.


3. `includes/marketplace.php`
-----------------------------

Main code purpose:
- Add missing schema pieces needed by the new marketplace flow
- Provide helper functions for milestone formatting

Main functions:

`marketplaceTableExists(mysqli $conn, string $table): bool`
- Checks if a table exists using `SHOW TABLES LIKE ...`

`marketplaceColumnExists(mysqli $conn, string $table, string $column): bool`
- Checks if a column exists using `SHOW COLUMNS ... LIKE ...`

`marketplaceEnsureQuery(mysqli $conn, string $sql): void`
- Runs raw SQL query

`ensureMarketplaceSchema(mysqli $conn): void`
- Uses a static `$done` flag so it only runs once per request
- Checks for missing columns and tables, then adds them
- Updates `orders`, `payments`, `chats`
- Creates `order_progress` if missing

`decodeMilestones(?string $milestoneJson): array`
- Converts milestone JSON from database into PHP array

`formatPaymentMode(?string $paymentMode): string`
- Returns readable label:
  - `Milestone payment`
  - `Full payment`

Why it matters:
- This is the support layer for your Fiverr-style workflow.
- It avoids manual DB edits when code runs on an older schema.


4. `includes/auth.php`
----------------------

Main code purpose:
- Start session
- Protect routes

- `redirectIfUser()`

Why it matters:
- Every protected page depends on these functions for access control.

6. `user/order-service.php`
---------------------------

Main code purpose:
- Start a client request for a selected service

Step-by-step logic:
- Requires DB and auth files
- Calls `userAuth()`
- Reads service ID from `$_GET['id']`
- Loads the selected service from `services` table
- Validates POST fields:
  - `short_desc`
  - `detailed_desc`
  - `message`
  - optional `client_budget`

When form submits:
- Inserts into `orders`:
  - `user_id`
  - `service_id`
  - `short_desc`
  - `detailed_desc`
  - `client_budget`
  - `offer_status = waiting`
- Gets new order ID with `mysqli_insert_id($conn)`
- Inserts first chat row into `chats`
- Redirects to `user/chat/inbox.php?order_id=...&created=1`

Why it matters:
- This page replaced the old direct payment-first order flow.
- It is now the entry point for negotiation.


7. `user/chat/inbox.php`
------------------------

Main code purpose:
- Let client talk to admin about a specific order
- Handle offer actions

Important variables:
- `$selected_order_id`
- `$orders`
- `$selectedOrder`
- `$messages`
- `$milestones`
- `$error`
- `$success`

Code flow:

A. Initial order loading
- Fetches all orders for the logged-in user
- If no `order_id` is passed, picks the first order automatically

B. POST handling
- Reads:
  - `message`
  - `action`
  - optional file upload
- Validates selected order belongs to current user

C. File upload logic
- Stores uploaded file in `uploads/chat/`
- Allows:
  - jpg
  - jpeg
  - png
  - zip

D. Offer action logic
- `accept_offer`
  - sets `orders.status = approved`
  - sets `orders.offer_status = accepted`
  - creates chat message saying client accepted

- `reject_offer`
  - sets `orders.status = rejected`
  - sets `orders.offer_status = rejected`
  - creates chat message saying client rejected

- `counter_offer`
  - reads `counter_price`
  - reads `counter_message`
  - builds a text message like:
    - `Counter offer $300.00: ...`
  - updates:
    - `offer_status = countered`
    - `status = pending`

E. Message insert
- Inserts new message into `chats` table
- Includes:
  - `order_id`
  - `sender_id`
  - `sender_role = user`
  - `message`
  - `file`

F. Render section
- Shows order selector
- Shows order summary / offer info
- Shows message history
- Shows offer action buttons when admin has sent an offer
- Shows payment button if offer was accepted

Why it matters:
- This is the main user-side negotiation screen.

9. `admin/chat/send.php`
------------------------

Main code purpose:
- Admin chat page for one order
- Admin can send both normal messages and custom offers

Important logic blocks:

A. Order loading
- Reads `order_id` from GET/POST
- Loads:
  - order data
  - user name
  - service title

B. File upload
- Saves attachment into `uploads/chat/`

C. Offer builder
- Triggered when `send_offer` exists in POST
- Reads:
  - `offer_price`
  - `payment_mode`
  - `milestone_title[]`
  - `milestone_amount[]`

D. Full-payment offer logic
- If payment mode = `full`
- stores price
- clears or ignores milestone plan

E. Milestone offer logic
- Loops through milestone arrays
- Builds milestone array like:
  - title
  - amount
- Sums milestone amounts
- Verifies milestone total equals full offer price
- Converts milestones to JSON using `json_encode(...)`

F. Update order
- Updates:
  - `offered_price`
  - `payment_mode`
  - `milestone_plan`
  - `offer_status = offered`
  - `status = pending`

G. Offer message creation
- Builds a readable chat message:
  - `Admin sent a full payment offer for $...`
  - or milestone version
- Appends custom note/message if provided

H. Chat insert
- Inserts admin message into `chats`

Why it matters:
- This is the main admin-side negotiation engine.

24. Practical Warning
---------------------

Because this project is procedural PHP, many pages directly mix:
- query logic
- form handling
- page rendering

So when editing a feature:
- first check POST handling
- then DB queries
- then redirects
- then UI rendering

This is important because one file often does everything in one place.
